## 소개
WeKora는 즉시 프로덕션에 투입할 수 있는 엔터프라이즈급 RAG 프레임워크로, 지능형 문서 이해와 검색 기능을 제공합니다. 이 시스템은 모듈형 설계를 채택하여 문서 이해, 벡터 저장, 추론 파일 등의 기능을 분리합니다.

## PipeLine
WeKnora는 문서를 처리하기 위해 여러 단계를 거칩니다: 삽입 -> 지식 추출 -> 인덱싱 -> 검색 -> 생성. 전체 흐름은 다양한 검색 방식을 지원합니다.

![](./images/pipeline2.jpeg)

사용자가 업로드한 숙박 거래 내역 pdf 파일을 예로 들어 데이터 흐름을 설명합니다:

### 1. 요청 수신 및 초기화
+ **요청 식별**: 시스템이 요청을 수신하고 고유한 `request_id=Lkq0OGLYu2fV`를 부여해 전체 처리 흐름을 추적합니다.
+ **테넌트 및 세션 검증**:
    - 시스템이 먼저 테넌트 정보를 검증합니다(ID: 1, Name: Default Tenant).
    - 이어서 지식베이스 QA(Knowledge QA) 요청을 처리하며, 해당 요청은 세션 `1f241340-ae75-40a5-8731-9a3a82e34fdd`에 속합니다.
+ **사용자 질문**: 사용자의 원 질문은 "**입실한 객실 타입은 무엇인가**"입니다.
+ **메시지 생성**: 시스템이 질문과 생성될 답변에 대한 메시지 레코드를 생성하며, ID는 각각 `703ddf09-...`와 `6f057649-...`입니다.

### 2. 지식베이스 QA 파이프라인 시작
시스템이 지식베이스 QA 서비스를 호출하고, 순서대로 실행될 전체 처리 파이프라인(Pipeline)을 정의합니다. 포함되는 9개 이벤트는 다음과 같습니다:
`[rewrite_query, preprocess_query, chunk_search, chunk_rerank, chunk_merge, filter_top_k, into_chat_message, chat_completion_stream, stream_filter]`

---

### 3. 이벤트 실행 상세
#### 이벤트 1: `rewrite_query` - 질문 재작성
+ **목적**: 더 정확한 검색을 위해 문맥을 결합해 사용자의 실제 의도를 이해합니다.
+ **작업**:
    1. 시스템이 현재 세션의 최근 20개 메시지를 조회(실제로는 8개)해 컨텍스트로 사용합니다.
    2. `deepseek-r1:7b` 로컬 LLM을 호출합니다.
    3. 모델이 대화 이력에서 질문자가 "Liwx"임을 분석하고, 원 질문 "입실한 객실 타입은 무엇인가"를 더 구체적으로 재작성합니다.
+ **결과**: 질문이 "**Liwx 이번 입실의 객실 타입은 무엇인가**"로 재작성됩니다.

#### 이벤트 2: `preprocess_query` - 질문 전처리
+ **목적**: 재작성된 질문을 토큰화하여 검색 엔진에 적합한 키워드 시퀀스로 변환합니다.
+ **작업**: 재작성된 질문을 토큰화합니다.
+ **결과**: 다음 키워드가 생성됩니다: "`개작성 필요 사용자 질문 입실 객실 제공 정보 입실자 Liwx 선택 객실 트윈룸 따라서 개작성 후 완전한 질문은 Liwx 이번 입실 객실`".

#### 이벤트 3: `chunk_search` - 지식 블록 검색
가장 핵심적인 **검색(Retrieval)** 단계로, 시스템은 두 번의 혼합 검색(Hybrid Search)을 수행합니다.

+ **첫 번째 검색(재작성된 완전 문장 사용)**:
    - **벡터 검색**:
        1. 임베딩 모델 `bge-m3:latest`로 문장을 1024차원 벡터로 변환합니다.
        2. PostgreSQL에서 벡터 유사도 검색을 수행해 관련 지식 블록(chunk) 2개를 찾습니다(ID: `e3bf6599-...`, `3989c6ce-...`).
    - **키워드 검색**:
        1. 동시에 키워드 검색을 수행합니다.
        2. 동일하게 2개의 지식 블록을 찾습니다.
    - **결과 병합**: 두 방식에서 얻은 4개의 결과(실제로는 2개가 중복)를 중복 제거해 최종 2개의 고유 지식 블록을 얻습니다.
+ **두 번째 검색(전처리된 키워드 시퀀스 사용)**:
    - 전처리된 키워드로 위의 **벡터 검색**과 **키워드 검색**을 반복합니다.
    - 최종적으로 동일한 2개의 지식 블록을 얻습니다.
+ **최종 결과**: 두 차례 검색과 병합을 거쳐 가장 관련성이 높은 2개 지식 블록을 확정하고, 답변 생성에 사용할 내용을 추출합니다.

#### 이벤트 4: `chunk_rerank` - 결과 재정렬
+ **목적**: 더 강력한 모델로 초기 검색 결과를 정밀하게 재정렬해 최종 답변 품질을 높입니다.
+ **작업**: 로그에 `Rerank model ID is empty, skipping reranking`가 기록됩니다. 이는 재정렬 단계가 설정되어 있지만 구체적인 재정렬 모델을 지정하지 않아 **해당 단계가 건너뛰어졌음**을 의미합니다.

#### 이벤트 5: `chunk_merge` - 블록 병합
+ **목적**: 인접하거나 관련 있는 지식 블록을 병합해 더 완전한 컨텍스트를 구성합니다.
+ **작업**: 시스템이 2개의 블록을 분석하고 병합을 시도합니다. 로그에 따르면 최종적으로 2개의 독립된 블록으로 유지되며 관련성 점수 순으로 정렬됩니다.

#### 이벤트 6: `filter_top_k` - Top-K 필터
+ **목적**: 가장 관련성이 높은 K개 결과만 유지해 불필요한 정보가 언어 모델에 영향을 주지 않게 합니다.
+ **작업**: 시스템이 Top-K=5로 설정합니다. 현재 블록이 2개뿐이므로 모두 통과합니다.

#### 이벤트 7 & 8: `into_chat_message` & `chat_completion_stream` - 답변 생성
**생성(Generation)** 단계입니다.

+ **목적**: 검색된 정보를 기반으로 자연스럽고 유창한 답변을 생성합니다.
+ **작업**:
    1. 시스템이 2개의 지식 블록, 사용자의 원 질문, 대화 이력을 결합해 하나의 Prompt를 구성합니다.
    2. 다시 `deepseek-r1:7b` LLM을 호출하고 **스트리밍(Stream)** 방식으로 답변을 생성합니다. 스트리밍 출력은 타이핑 효과를 구현해 사용자 경험을 향상합니다.

#### 이벤트 9: `stream_filter` - 스트림 출력 필터
+ **목적**: 모델이 생성하는 실시간 텍스트를 후처리해 불필요한 특수 표식이나 내용을 제거합니다.
+ **작업**:
    - 시스템이 `<think>`와 `</think>` 같은 내부 표식을 제거하는 필터를 설정합니다.
    - 로그에 따르면 모델의 첫 토큰은 `<think> 따라`였고, 필터가 `<think>`를 제거한 뒤 "따라" 이후의 내용만 전달합니다.

### 4. 완료 및 응답
+ **출처 전송**: 답변 생성과 동시에, 근거가 되는 2개의 지식 블록을 "참고 내용"으로 프런트엔드에 전송해 사용자가 출처를 확인할 수 있게 합니다.
+ **메시지 업데이트**: 모델이 답변을 완성하면, 이전에 생성된 메시지 레코드(ID: `6f057649-...`)에 최종 답변을 업데이트합니다.
+ **요청 종료**: 서버가 `200` 성공 상태 코드를 반환하며, 질문부터 답변까지의 전체 흐름이 종료됩니다.

### 요약
이 로그는 전형적인 RAG 흐름을 완전하게 기록합니다. 시스템은 **질문 재작성**과 **전처리**로 사용자 의도를 정확히 파악하고, **벡터+키워드 혼합 검색**으로 지식베이스에서 관련 정보를 찾습니다. **재정렬**은 건너뛰었지만 **병합**과 **필터링**을 수행했으며, 최종적으로 검색된 지식을 컨텍스트로 LLM이 유창하고 정확한 답변을 **생성**합니다. **스트림 필터링**으로 출력의 순도를 보장합니다.

## 문서 파싱 및 분할
코드는 gRPC로 통신하는 독립 마이크로서비스를 구현하며, 문서 내용을 깊이 있게 파싱하고 분할해 멀티모달 정보를 추출합니다. 이는 "비동기 처리" 단계의 핵심 실행자입니다.

### **전체 아키텍처**
Python 기반 gRPC 서비스로, 파일(또는 URL)을 수신해 이후 처리(예: 벡터화)에 사용할 수 있는 구조화된 텍스트 블록(Chunks)으로 파싱합니다.

+ `server.py`: 서비스 진입점 및 네트워크 레이어. 멀티프로세스/멀티스레드 gRPC 서버를 띄우고 Go 백엔드 요청을 처리해 파싱 결과를 반환합니다.
+ `parser.py`: **Facade(퍼사드) 패턴**. `Parser` 클래스를 통해 PDF, DOCX, Markdown 등 다양한 파서를 추상화합니다. 외부 호출자는 `Parser`와만 상호작용합니다.
+ `base_parser.py`: 파서 베이스 클래스. 모든 파서가 공유하는 핵심 로직과 추상 메서드를 정의합니다. 텍스트 분할, 이미지 처리, OCR, 이미지 캡션 생성 등 가장 복잡한 기능이 여기 있습니다.

---

### **상세 작업 흐름**
Go 백엔드가 비동기 작업을 시작하면 파일 내용과 설정 정보를 담아 이 Python 서비스로 gRPC 호출을 수행합니다. 전체 흐름은 다음과 같습니다.

#### **1단계: 요청 수신 및 분배 (`server.py` & `parser.py`)**
1. **gRPC 서비스 진입점(`server.py: serve`)**:
    - 서비스는 `serve()`로 시작됩니다. 환경 변수(`GRPC_WORKER_PROCESSES`, `GRPC_MAX_WORKERS`)에 따라 **멀티프로세스/멀티스레드** 서버를 실행해 CPU를 최대한 활용하고 동시 처리 능력을 높입니다.
    - 각 워커 프로세스는 지정 포트(예: 50051)를 리슨하며 요청을 대기합니다.
2. **요청 처리(`server.py: ReadFromFile`)**:
    - Go 백엔드가 `ReadFromFile` 요청을 보내면 워커 프로세스가 이를 수신합니다.
    - 요청 파라미터를 파싱합니다:
        * `file_name`, `file_type`, `file_content`: 파일 기본 정보 및 바이너리 내용
        * `read_config`: `chunk_size`, `chunk_overlap`, `enable_multimodal`, `storage_config`, `vlm_config` 등 파싱 설정을 담은 복합 객체
    - 설정을 `ChunkingConfig` 데이터 객체로 통합합니다.
    - 핵심 단계로 `self.parser.parse_file(...)`을 호출해 파싱 작업을 `Parser` 퍼사드에 위임합니다.
3. **파서 선택(`parser.py: Parser.parse_file`)**:
    - `Parser`는 `get_parser(file_type)`을 호출합니다.
    - 파일 타입(예: `'pdf'`)에 따라 `self.parsers`에서 해당 파서를 찾습니다(예: `PDFParser`).
    - 해당 `PDFParser`를 **인스턴스화**하고 `ChunkingConfig` 등 모든 설정을 생성자에 전달합니다.

#### **2단계: 핵심 파싱 및 분할(`base_parser.py`)**
이 단계는 **정보의 컨텍스트 완전성과 원래 순서를 어떻게 보장하는지**에 대한 핵심을 다룹니다.

`base_parser.py`에 따르면 **최종 분할된 Chunk의 텍스트, 표, 이미지는 원문 문서의 등장 순서대로 저장됩니다**.

이는 `BaseParser` 내 여러 메서드가 정교하게 협력하기 때문입니다. 흐름을 단계별로 살펴보면 다음과 같습니다.

1. **단계 1: 통합 텍스트 스트림 생성(`pdf_parser.py`)**:
    - `parse_into_text`에서 PDF를 **페이지 단위로** 처리합니다.
    - 각 페이지 내부에서 (표 외 텍스트 -> 표 -> 이미지 자리표시자) 순으로 내용을 **하나의 긴 문자열**(`page_content_parts`)로 합칩니다.
    - **핵심**: 이 순서가 문자 단위로 완벽하진 않더라도 **같은 페이지의 내용이 함께 유지되고**, 대체로 위에서 아래로 읽는 순서를 따릅니다.
    - 모든 페이지 내용을 `"\n\n--- Page Break ---\n\n"`로 연결해 **텍스트/표/이미지 자리표시자를 포함한 단일 정렬 텍스트(`final_text`)**를 만듭니다.
2. **단계 2: 원자화 및 보호(`_split_into_units`)**:
    - `final_text`를 `_split_into_units`에 전달합니다.
    - 이 메서드는 **Markdown 표**와 **Markdown 이미지 자리표시자**를 **분리 불가능한 원자 단위(atomic units)**로 인식하도록 정규식을 사용합니다.
    - 표/이미지와 그 사이의 일반 텍스트를 `final_text` 내 **원래 순서대로** 리스트(`units`)로 분할합니다.
    - **결과**: `['텍스트', '![...](...)', '다른 텍스트', '|...|...|\n|---|---|\n...', '추가 텍스트']`처럼 원본 순서가 보존된 리스트가 됩니다.
3. **단계 3: 순서 기반 분할(`chunk_text`)**:
    - `chunk_text`가 **순서가 보장된 `units` 리스트**를 입력으로 받습니다.
    - 리스트의 각 단위를 **순서대로** 순회하며 `current_chunk`에 추가합니다.
    - `chunk_size`에 가까워지면 블록을 저장하고, 새 블록을 시작합니다(일부 오버랩 포함 가능).
    - **핵심**: `chunk_text`가 `units`를 **순서대로** 처리하므로 표/텍스트/이미지의 상대적 순서가 절대 바뀌지 않습니다. 문서에서 먼저 등장한 표는 반드시 앞쪽 Chunk에 포함됩니다.
4. **단계 4: 이미지 정보 부가(`process_chunks_images`)**:
    - 텍스트 분할이 완료되면 `process_chunks_images`가 호출됩니다.
    - 각 Chunk에서 이미지 자리표시자를 찾아 AI 처리를 수행합니다.
    - 처리 결과(영구 URL, OCR 텍스트, 이미지 설명)를 해당 Chunk의 `.images`에 추가합니다.
    - **핵심**: 이 과정은 Chunk의 순서나 `.content`를 변경하지 않고, 기존 Chunk에 정보만 추가합니다.

#### **3단계: 멀티모달 처리(활성화 시, `base_parser.py`)**
`enable_multimodal`이 `True`이면 텍스트 분할 이후 멀티모달 처리 단계로 들어갑니다.

1. **병렬 작업 시작(`BaseParser.process_chunks_images`)**:
    - `asyncio`로 모든 텍스트 블록의 이미지를 **병렬 처리**해 효율을 높입니다.
    - 각 `Chunk`마다 `process_chunk_images_async` 비동기 작업을 생성합니다.
2. **단일 블록 이미지 처리(`BaseParser.process_chunk_images_async`)**:
    - **이미지 참조 추출**: `extract_images_from_chunk` 정규식으로 `![alt text](image.png)` 형태의 이미지 참조를 찾습니다.
    - **이미지 영속화**: 각 이미지를 `download_and_upload_image`로 병렬 처리합니다.
        * 원본 위치(PDF 내부, 로컬 경로, 원격 URL)에서 이미지 데이터를 가져옵니다.
        * 이미지를 **객체 스토리지(COS/MinIO)**에 업로드해, 임시 참조를 영구 URL로 변환합니다.
        * 영구 URL과 이미지 객체(PIL Image)를 반환합니다.
    - **AI 병렬 처리**: 업로드된 이미지들을 `process_multiple_images`로 전달합니다.
        * `asyncio.Semaphore`로 동시 처리 개수(예: 5장)를 제한해 메모리 사용과 API 레이트 리밋을 제어합니다.
        * 각 이미지마다 `process_image_async`를 호출합니다.
3. **단일 이미지 처리(`BaseParser.process_image_async`)**:
    - **OCR**: `perform_ocr`로 PaddleOCR 같은 엔진을 사용해 이미지 텍스트를 인식합니다.
    - **이미지 캡션**: `get_image_caption`이 이미지를 Base64로 변환해 VLM에 전달하고, 이미지 설명을 생성합니다.
    - 결과는 `(ocr_text, caption, 영구 URL)`입니다.
4. **결과 집계**:
    - 모든 이미지 처리가 완료되면, 영구 URL, OCR 텍스트, 이미지 설명이 포함된 구조화 정보를 해당 `Chunk`의 `.images`에 추가합니다.

#### **4단계: 결과 반환(`server.py`)**
1. **데이터 변환(`server.py: _convert_chunk_to_proto`)**:
    - `parser.parse_file`이 모든 처리된 `Chunk` 리스트(`ParseResult`)를 반환합니다.
    - `ReadFromFile`은 `_convert_chunk_to_proto`로 Python `Chunk`(이미지 정보 포함)를 gRPC Protobuf 메시지로 변환합니다.
2. **응답 반환**:
    - gRPC 서버가 분할 및 멀티모달 정보를 포함한 `ReadResponse`를 호출자(Go 백엔드)에게 반환합니다.

이로써 Go 백엔드는 구조화되고 정보가 풍부한 문서 데이터를 확보하고, 다음 단계인 벡터화와 인덱싱 저장을 진행할 수 있습니다.

## QA
### 질문 1: 검색 과정에서 두 번의 혼합 검색을 수행하는 목적은 무엇이며, 두 검색의 차이는 무엇인가?
좋은 관찰입니다. 두 번의 혼합 검색은 **정확도와 재현율을 최대화**하기 위한 것으로, 본질적으로 **Query Expansion과 다중 전략 검색**의 조합입니다.

#### 목적
두 가지 형태의 쿼리(재작성된 원문 문장 vs. 토큰화된 키워드 시퀀스)로 검색하면 각각의 장점을 결합할 수 있습니다:

+ **의미 검색의 깊이**: 완전한 문장 검색은 `bge-m3` 같은 벡터 모델이 문장 전체 의미를 잘 이해해 의미적으로 가까운 지식 블록을 찾습니다.
+ **키워드 검색의 범위**: 토큰화된 키워드는 표현이 다르더라도 핵심 키워드가 포함되면 매칭될 가능성을 높입니다. 이는 BM25 같은 전통 키워드 매칭에 특히 효과적입니다.

즉, **같은 질문을 다른 방식으로 두 번 묻고 결과를 합쳐** 중요한 지식이 누락되지 않게 합니다.

#### 두 번의 검색 차이
가장 큰 차이는 **입력 쿼리 텍스트(Query Text)**입니다.

1. **첫 번째 혼합 검색**
    - **입력**: `rewrite_query` 이벤트 이후 생성된 **문법적으로 완전한 자연어 질문**
    - **로그 근거**:

```plain
INFO [2025-08-29 09:46:36.896] [request_id=Lkq0OGLYu2fV] knowledgebase.go:266[HybridSearch] | Hybrid search parameters, knowledge base ID: kb-00000001, query text: 재작성할 사용자 질문은 “입실한 객실 타입은 무엇인가”입니다. 제공된 정보에 따르면 입실자 Liwx가 선택한 객실은 트윈룸입니다. 따라서 재작성된 완전한 질문은 “Liwx 이번 입실의 객실 타입은 무엇인가”입니다.
```

2. **두 번째 혼합 검색**
    - **입력**: `preprocess_query` 처리 후 생성된 **공백으로 분리된 키워드 시퀀스**
    - **로그 근거**:

```plain
INFO [2025-08-29 09:46:37.257] [request_id=Lkq0OGLYu2fV] knowledgebase.go:266[HybridSearch] | Hybrid search parameters, knowledge base ID: kb-00000001, query text: 개작성 필요 사용자 질문 입실 객실 제공 정보 입실자 Liwx 선택 객실 트윈룸 따라서 개작성 후 완전한 질문은 Liwx 이번 입실 객실
```

최종적으로 시스템은 두 검색 결과를 중복 제거 후 병합합니다(로그상 매번 2개 결과가 나오고, 병합 후에도 2개로 유지됨). 이를 통해 더 신뢰도 높은 지식 집합을 확보해 답변 생성에 사용합니다.

### 질문 2: 재정렬 모델 분석
Reranker(재정렬)는 RAG 분야에서 매우 고급 기술이며, 동작 원리와 적용 시나리오가 뚜렷하게 다릅니다.

간단히 말해, **전용 판별 모델** -> **LLM 기반 판별** -> **LLM 내부 정보의 심층 활용**으로의 진화를 의미합니다.

아래는 상세한 차이입니다.

#### 1. Normal Reranker(일반 재정렬기 / Cross-Encoder)
가장 전통적이고 널리 쓰이는 재정렬 방식입니다.

+ **모델 유형**: **Sequence Classification Model**. 본질적으로 **Cross-Encoder**이며 BERT, RoBERTa 등 양방향 인코더 기반입니다. `BAAI/bge-reranker-base/large/v2-m3`가 대표적입니다.
+ **동작 원리**:
    1. **Query**와 **Passage**를 하나의 입력 시퀀스로 결합합니다. 예: `[CLS] what is panda? [SEP] The giant panda is a bear species endemic to China. [SEP]`.
    2. 모델의 Self-Attention이 쿼리와 문서의 단어를 함께 분석해 **세밀한 상호작용**을 계산합니다.
    3. 최종적으로 **단일 점수(Logit)**를 출력하며, 점수가 높을수록 관련성이 높습니다.
+ **핵심 특성**:
    - **장점**: 쿼리-문서 간 깊은 상호작용으로 **정확도가 매우 높아** Reranker 성능의 기준이 됩니다.
    - **단점**: **속도가 느림**. 검색 결과가 100개면 100번의 고비용 추론이 필요합니다.

#### 2. LLM-based Reranker(LLM 기반 재정렬)
일반 LLM의 능력을 활용해 재정렬을 수행합니다.

+ **모델 유형**: **Causal Language Model**. GPT, Llama, Gemma 같은 생성형 LLM이 해당됩니다. `BAAI/bge-reranker-v2-gemma`가 예시입니다.
+ **동작 원리**:
    1. 직접 점수를 출력하지 않고, 재정렬을 **질의응답/텍스트 생성 작업**으로 변환합니다.
    2. 프롬프트를 구성합니다. 예: `"Given a query A and a passage B, determine whether the passage contains an answer to the query by providing a prediction of either 'Yes' or 'No'. A: {query} B: {passage}"`.
    3. LLM이 생성하는 마지막 “Yes”의 확률을 관찰합니다.
    4. **“Yes” 생성 확률(Logit)**을 관련성 점수로 사용합니다.
+ **핵심 특성**:
    - **장점**: LLM의 **의미 이해, 추론, 세계 지식**을 활용해 복잡한 쿼리에서 더 나은 성능을 기대할 수 있습니다.
    - **단점**: 모델 크기에 따라 비용이 크며, **프롬프트 설계에 크게 의존**합니다.

#### 3. LLM-based Layerwise Reranker(LLM 레이어 기반 재정렬)
2번 방식의 확장형으로, 더 전위적이고 복잡한 방법입니다.

+ **모델 유형**: 동일하게 **Causal Language Model**. 예: `BAAI/bge-reranker-v2-minicpm-layerwise`.
+ **동작 원리**:
    1. 입력은 2번과 동일하게 “Yes/No” 프롬프트를 사용합니다.
    2. 차이는 **점수 추출 방식**입니다. 마지막 레이어만 보는 대신, **여러 중간 레이어**에서 “Yes”의 Logit을 추출합니다.
    3. `cutoff_layers=[28]`는 28번째 레이어 출력을 사용하라는 의미입니다.
    4. 여러 레이어 점수를 평균/조합해 더 견고한 관련성 판단을 얻습니다.
+ **핵심 특성**:
    - **장점**: 더 **풍부하고 전반적인 관련성 신호**를 얻어 마지막 레이어만 볼 때보다 높은 정확도를 기대할 수 있습니다.
    - **단점**: **복잡도가 가장 높고** `trust_remote_code=True`처럼 모델 수정이 필요하며 비용도 큽니다.

#### 요약 비교
| 특성 | 1. Normal Reranker | 2. LLM-based Reranker | 3. LLM-based Layerwise Reranker |
| :--- | :--- | :--- | :--- |
| **기반 모델** | Cross-Encoder(BERT 등) | Causal LM(Gemma 등) | Causal LM(MiniCPM 등) |
| **동작 원리** | Query/Passage 깊은 상호작용 후 점수 출력 | Yes/No 예측에서 “Yes” 확률을 점수로 사용 | 여러 레이어의 “Yes” 확률을 조합 |
| **출력** | 단일 점수 | 단일 점수(마지막 레이어) | 다중 점수(여러 레이어) |
| **장점** | **속도-정확도 균형**이 가장 좋고 안정적 | LLM의 추론 능력 활용 | 이론상 가장 높은 정확도 |
| **단점** | 벡터 검색 대비 느림 | 비용이 크고 프롬프트 의존 | **가장 높은 복잡도와 비용** |
| **추천 시나리오** | **대부분의 프로덕션 환경에 적합** | 품질 극대화가 필요하고 자원이 충분할 때 | 연구/최신 성능 추구 환경 |

#### 사용 제안
1. **초기 단계**: `BAAI/bge-reranker-v2-m3` 같은 `Normal Reranker`부터 시작하는 것을 권장합니다. 성능이 뛰어나고 통합이 쉽습니다.
2. **심화 단계**: 정교한 추론이 필요한 쿼리에서 성능이 부족하고 GPU 자원이 충분하다면 `LLM-based Reranker`를 고려합니다.
3. **연구 단계**: `Layerwise Reranker`는 연구자나 SOTA 성능을 추구하는 전문가에게 적합합니다.

핵심 제약: 제공된 문서만으로 엄격히 답변하며 자체 지식을 사용하지 않는다
알 수 없는 경우 처리: 문서에 충분한 정보가 없으면 "보유한 자료로는 이 প্রশ্ন에 답할 수 없습니다"라고 안내한다
